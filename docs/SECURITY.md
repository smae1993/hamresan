# مدل امنیتی

## تضمین‌های فعلی

- هر انتقال به قبول صریح گیرنده نیاز دارد.
- Token یک‌بارمصرف، `transferId`، شناسهٔ فرستنده و IP با نشست پذیرفته‌شده تطبیق داده می‌شوند.
- نام فایل و پوشه پاک‌سازی می‌شود و `..` یا Separatorهای مسیر قابل استفاده نیستند.
- فایل ابتدا با نام موقت نوشته می‌شود.
- SHA-256 فایل قبل از Rename نهایی بررسی می‌شود.
- فایل موجود بازنویسی نمی‌شود و نام Collision-safe تولید می‌گردد.
- Header، تعداد فایل و حجم کل محدود شده‌اند.

## تضمین‌های فعلاً موجود نیست

نسخهٔ 1 پروتکل محرمانگی سرتاسری و احراز هویت رمزنگاری‌شدهٔ دستگاه را تضمین نمی‌کند. SHA-256 در این نسخه برای تشخیص خرابی انتقال است و جایگزین MAC یا Encryption نیست. UI و مستندات انتشار نباید تا پیش از تکمیل مرحلهٔ زیر ادعای «رمزنگاری فعال» داشته باشند.

## مرحلهٔ امنیتی بعدی

- هویت Ed25519 پایدار برای هر نصب
- تبادل کلید Ephemeral با X25519
- مشتق‌سازی کلید نشست با HKDF-SHA256
- فریم‌های AEAD با AES-GCM یا ChaCha20-Poly1305
- Pairing با QR یا کد کوتاه و تأیید Fingerprint
- پذیرش خودکار فقط برای Public Keyهای مورد اعتماد
- جلوگیری از Replay با Nonce و Session Counter

پیاده‌سازی الگوریتم رمزنگاری اختصاصی ممنوع است؛ فقط Primitiveهای کتابخانهٔ ممیزی‌شده قابل استفاده‌اند.

## گزارش آسیب‌پذیری

تا زمان تعریف کانال خصوصی، مشکل امنیتی را در Issue عمومی همراه فایل یا دادهٔ واقعی کاربران منتشر نکنید. ابتدا با مالک مخزن هماهنگ کنید.
